# Session Expiration and CLI Token Lifetime

## Status

- Proposed and approved for implementation.

## Context

The system currently supports session lifecycle states for creation, live streaming, normal completion, and failure, but it does not model time-based session closure. CLI connection tokens are also signed using a global environment-based lifetime rather than a backend-defined policy tied to session behavior.

This specification introduces explicit session expiration rules, a distinct `expired` status, and a fixed CLI token lifetime policy.

## Goals

- Add a backend-owned session expiration model.
- Start the session expiration clock only after the first accepted `session.start` event.
- Reject CLI access for sessions that have expired.
- Introduce a first-class `expired` session status.
- Remove `TOKEN_EXPIRY_SECONDS` and stop relying on env configuration for CLI token lifetime.
- Show token lifetime guidance in the CLI setup dialog.

## Non-Goals

- Changing the CLI websocket protocol unless required for compatibility.
- Adding token refresh flows.
- Auto-extending sessions on reconnect.
- Introducing per-user or per-session configurable durations in this change.

## Definitions

- Session duration: `1h`
- CLI token duration: `1h 30m`
- Session start event: the first accepted `session.start` message from the CLI for a given session

## Session Lifecycle Rules

### Session creation

When a session is created:

- `status = "draft"`
- `startedAt = null`
- `endedAt = null`
- `expiresAt = null`

The session is not considered expiring or expired before it has started.

### First accepted `session.start`

When the backend accepts the first `session.start` for a session:

- if `startedAt` is `null`, set `startedAt` to the event timestamp
- if `expiresAt` is `null`, set `expiresAt = startedAt + 1h`
- set `status = "live"`

This operation is one-time initialization. Reconnects or repeated `session.start` messages must not extend or reset the expiration window.

### Normal completion

When a session ends normally:

- `status = "ended"`
- `endedAt` is set to the terminal event timestamp

### Failure

When a session ends because of an error:

- `status = "failed"`
- `endedAt` is set to the terminal event timestamp

### Expiration

A session is expired when all of the following are true:

- `startedAt` is not `null`
- `expiresAt` is not `null`
- current time is greater than or equal to `expiresAt`

When expiration is detected, the session must be treated as closed:

- `status = "expired"`
- `endedAt` is set if not already set
- no further CLI connections are accepted for that session
- no further CLI events are accepted for that session

## Session Status Model

The valid session statuses become:

- `draft`
- `live`
- `ended`
- `failed`
- `expired`

`expired` is a terminal state.

## CLI Token Rules

### Token generation

CLI tokens are generated by the backend with a fixed lifetime of `1h 30m` from token issuance.

This duration is independent from the session expiration window.

### Token signing

Token signing must not read a global expiry duration from environment variables.

Instead:

- `TOKEN_SECRET` remains required
- token expiration is passed explicitly to the signing function
- `TOKEN_EXPIRY_SECONDS` is removed everywhere

### Rationale

This allows users time to generate a token and connect the CLI, while keeping the actual session expiration window tied to real session start.

## CLI Access Enforcement

### Before websocket acceptance

On CLI websocket authentication / session validation:

- verify the token signature and token `exp`
- verify the token grants access to the target session
- reject the connection if the session is already expired
- reject the connection if the session is already in a closed terminal state where new CLI traffic should not be accepted

At minimum, expired sessions must be rejected.

### During event ingestion

Before persisting CLI-originated events:

- check whether the session has expired
- if expired, transition the session to `expired` if needed
- reject the event

This second guard prevents already-open websocket connections from continuing to send events past expiration.

## Data Model Changes

### Shared session contract

Add `expiresAt: Date | null` to the shared session schema and all parsed session payloads.

### Database

Add nullable `expires_at` to `sessions`.

Initial row values:

- `created_at`: existing behavior
- `started_at`: nullable
- `ended_at`: nullable
- `expires_at`: nullable

## UI Requirements

### CLI setup dialog

`web/src/components/sessions/cli-setup-dialog.tsx` must communicate:

- the generated CLI token is valid for `1 hour 30 minutes`
- the session expiration clock starts only after the CLI starts the session
- once a session expires, it is closed and cannot accept more CLI connections
- users should regenerate the token only if the token expired; if the session itself expired, they need a new session

If the API returns token expiration metadata, the dialog may also show the exact expiration timestamp.

### Session UI

Session status UI must support the `expired` status distinctly from `ended` and `failed`.

## Protocol Notes

No protocol change is required for the CLI to remain compatible.

The existing websocket message types remain:

- `session.start`
- `session.ready`
- `transcript.partial`
- `transcript.final`
- `session.stop`

The backend may continue to return `session.error` when rejecting expired or closed sessions.

## Implementation Plan

### Phase 1: Contracts and schema

1. Add `expired` to session status values.
2. Add nullable `expiresAt` to the shared session contract.
3. Add nullable `expires_at` to the `sessions` table schema.
4. Update Drizzle SQL and snapshot metadata.

Expected files:

- `web/src/lib/contracts/session.ts`
- `web/src/server/db/schema.ts`
- `web/drizzle/0000_reflective_genesis.sql`
- `web/drizzle/meta/0000_snapshot.json`

### Phase 2: Token lifetime changes

1. Remove `TOKEN_EXPIRY_SECONDS` usage from token signing.
2. Change token signing to accept explicit expiry input.
3. Set CLI token expiration to `now + 1h 30m` in session token creation.
4. Remove `TOKEN_EXPIRY_SECONDS` from documented env files.

Expected files:

- `web/src/server/token.ts`
- `web/src/server/api/routers/session.ts`
- `web/.env.example`
- optionally local dev env files that still reference the removed variable

### Phase 3: Session expiration lifecycle

1. On first accepted `session.start`, initialize `startedAt` and `expiresAt`.
2. Add helpers to determine whether a session is expired.
3. Transition expired sessions to `expired` when expiration is detected.
4. Prevent further event ingestion for expired sessions.

Expected files:

- `web/src/server/api/session-event-ingest.ts`
- supporting helpers if extracted

### Phase 4: CLI access enforcement

1. Reject websocket connections for expired sessions.
2. Return a clear auth/session error message for expired sessions.
3. Keep session validation consistent between websocket acceptance and event ingestion.

Expected files:

- `web/src/server/ws-ingest.ts`

### Phase 5: UI updates

1. Support `expired` in session status display logic.
2. Update live/session views to reflect the new terminal state.
3. Update CLI setup copy to explain token lifetime and session expiration behavior.
4. Optionally show exact token expiration time if returned by the API.

Expected files:

- `web/src/lib/session-ui.ts`
- `web/src/components/sessions/session-live-view.tsx`
- `web/src/components/sessions/cli-setup-dialog.tsx`

### Phase 6: Verification

1. Type-check web code.
2. Lint changed web files.
3. Build the web app.
4. Manually verify:
   - creating a session leaves `expiresAt = null`
   - generating a token succeeds and UI shows the new guidance
   - first `session.start` sets `startedAt` and `expiresAt`
   - expired sessions reject new CLI connections
   - expired sessions reject further events on an already-open connection
   - session status renders as `expired`

Suggested commands:

- `cd web && bun run lint`
- `cd web && bunx tsc --noEmit`
- `cd web && bun run build`
